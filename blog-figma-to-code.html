From Figma to Code: How I Bridge Design & Development

By Ericka Mae Gavino · February 10, 2026 · 6 min read

Most people treat design and development as two separate phases. I treat them as one continuous conversation — and it changed the way I build everything.

The Problem With the “Handoff” Mentality

In many teams, the workflow looks like this: a designer creates a stunning mockup in Figma, exports assets, writes a few notes in a comment thread, and hands it off. The developer then spends the next few days trying to interpret spacing values, argue about whether a font is 500 or 600 weight, and rebuild components they don't fully understand.

I've been on both sides of that wall. And the truth is — the wall itself is the problem.

When I design a component in Figma, I'm already thinking about how it'll be built. I name layers the way I'd name CSS classes. I use Auto Layout the same way I'd think about Flexbox. I don't just design a card — I design a reusable card component with states.

My Figma Setup Before I Write a Single Line of Code

Before opening VS Code, I run through this checklist:

  • Design tokens first. I set up a local styles library in Figma — colours, type scale, spacing units. These map directly to CSS custom properties later.
  • Auto Layout everywhere. If a container won't be constrained in the browser, it shouldn't be constrained in Figma. Auto Layout forces me to think in terms of flex and grid direction from the start.
  • Break the design into components. Buttons, cards, nav items, form inputs — each one has its own Figma component. In code, each one has its own CSS class.
  • Export nothing until the design is final. Exporting too early creates versioning chaos. I finish the design, then write all the HTML/CSS at once.

Translating Visual Decisions into Code

Here's the most practical part. When I look at a Figma frame and start translating it:

Spacing — I inspect the padding and gap values in Figma and immediately map them to my CSS spacing scale. If Figma shows 24px padding, that becomes padding: 24px or a custom property like var(--space-6).

Typography — I never eyeball font sizes. Figma's inspect panel gives exact values. I set them in rem so they scale with user preferences — 1.5rem for 24px base.

Colour — Every hex I use in CSS comes directly from the Figma styles panel. No approximations. No "it looks close enough."

Shadows and borders — Figma's drop shadow values (x, y, blur, spread, colour) translate directly to CSS box-shadow. I copy them exactly.

The Part Most Tutorials Skip: Responsive Behaviour

A Figma frame is a fixed canvas. The browser is not. The biggest bridge I build between design and code isn't about pixels — it's about thinking through how a layout breaks at 768px, 480px, and everything in between.

My rule: I design at 1440px desktop width, then before writing any CSS, I sketch (even on paper) what a 375px mobile layout should look like. Not a perfect mockup — just a rough sense of what stacks, what hides, and what scales. That sketch prevents 80% of responsive bugs before they happen.

What This Workflow Actually Saves

Doing both design and development in my own projects has taught me that the best code often comes from a designer's mind. Knowing why something looks the way it does — the intention behind a colour choice, a border radius, a hover animation — means I can implement it faithfully and make informed decisions when the design doesn't account for an edge case.

If you're a front-end developer who hasn't spent time in Figma, I'd genuinely encourage you to try it — not just as a viewer, but as a builder. And if you're a designer who hasn't written HTML and CSS, the same applies in reverse.

The best digital products I've seen come from people who don't fully identify as one or the other.
