Designing for Accessibility: What Real Projects Taught Me

By Ericka Mae Gavino · February 1, 2026 · 8 min read

Accessibility is one of those things that sounds optional until you realise it never was — it just wasn't enforced. Here's what years of building real products taught me about making the web actually work for everyone.

The Mindset Shift That Changed Everything

For a long time I thought of accessibility as a compliance requirement — a checklist that large companies worry about for legal reasons. Screen reader support, alt text on images, skip links. Check, check, check.

The shift happened when I started thinking about accessibility not as a set of rules for edge cases, but as a measure of how well a product works for all of its users — including the people who are tired, distracted, using an old phone, in bright sunlight, or navigating with a keyboard because their mouse died.

That's when the checklist became a design philosophy.

The Basics That Make the Biggest Difference

Not all accessibility improvements are equal. These four have the widest impact relative to the effort they require:

1. Colour contrast. WCAG AA requires a contrast ratio of 4.5:1 for normal text and 3:1 for large text. I use Figma's built-in contrast checker and browser dev tools to verify this before committing to a palette. Low contrast text doesn't just fail audits — it's genuinely hard to read for a huge proportion of your users, not just those with visual impairments.

2. Semantic HTML. The single highest-leverage accessibility improvement you can make is using the right element for the job. <button> instead of <div>. <nav> instead of <div class="nav">. <main>, <header>, <article>, <section>. These elements carry built-in meaning that screen readers and browsers rely on. You get keyboard support, focus management, and document structure for free.

3. Keyboard navigation. Tab through your interface before shipping it. Every interactive element — links, buttons, form inputs, modals — should be reachable and operable without a mouse. Focus styles should be visible (don't remove outline without replacing it). Modals should trap focus while open and return it to the trigger when closed.

4. Alternative text. Every image that conveys information needs a descriptive alt attribute. Every decorative image should have alt="" so screen readers skip it. The rule of thumb: if you removed the image and replaced it with the alt text, would the user still understand the content? If yes, you've written good alt text.

ARIA: Use It Less Than You Think You Should

ARIA (Accessible Rich Internet Applications) attributes let you describe the role, state, and properties of custom UI components. They're powerful — and dangerously easy to misuse.

The first rule of ARIA is: don't use ARIA if you can use a native HTML element instead. A <button> doesn't need role="button". An <input type="checkbox"> doesn't need aria-checked. Overusing ARIA creates false signals that confuse assistive technology.

Where ARIA genuinely earns its place: custom components that have no native HTML equivalent. A custom dropdown menu, a tab panel, a live notification region, a data grid. In those cases, ARIA provides the semantic layer that native HTML can't.

Designing Accessible Colour Palettes

One practical thing I learned from designing dark-themed interfaces: it's easy to pass contrast for large bold headings and fail it for body copy. The mistake is checking only your primary text colour against your primary background. You need to check every combination — placeholder text on inputs, caption text on cards, inactive nav items.

I now build colour palettes with accessibility in mind from the start, not as a retrofit. The question isn't "does this look good?" — it's "does this look good AND pass contrast for the text sizes where I'll use it?"

Testing Beyond Dev Tools

Automated tools like Lighthouse and axe catch about 30% of accessibility issues. The rest require human judgement. My testing process for any interface:

  • Run Lighthouse accessibility audit and fix all errors
  • Tab through the entire page with only the keyboard
  • Enable a screen reader (VoiceOver on Mac, NVDA on Windows) and navigate the page as a first-time user would
  • Test at 200% browser zoom — does the layout still work?
  • Test on a slow 3G connection — do loading states communicate clearly?

Accessibility Is Good Design

Every accessibility improvement I've made has also improved the experience for users without disabilities. Clear contrast benefits everyone reading in sunlight. Keyboard navigation helps power users. Good alt text improves SEO. Semantic HTML makes code easier to maintain.

Accessibility isn't a constraint on good design — it's a measure of it. The most inclusive interfaces are usually the most intuitive ones. That's not a coincidence.
